## LoopLlama and the tragedy of computer UX

Mouse as an anti-pattern. A computer mouse is a great invention for certain
kinds of tasks: things that, by their nature, inherently involve actual
pointing, positioning, dragging. But they an absurd way to issue routine
commands to a machine -- akin to a caveman pointing at the stone he wants.
Basic navigation, frequent operations -- none of it is a good fit for a tool
that requires you to grab, position, click. That kind of UX is not efficient
as it relates to time or ergonomics. It is also less repeatable, scriptable,
automatable, and scalable. The keyboard works better for that. Fast, literally
alread at hand.

Keyboard modifiers as an anti-pattern. Even applications with strong key
binding support tend to go astray because they take for granted the
conventions inherited from the largest operating systems. Key binding schemes
tend to quickly because awkward and difficult to remember beyond the basics.
Do I need CMD, CTRL, or SHIFT for that one? This may seem like a mere rehash
of classic vim vs emacs debates on the merits of chordal key bindings --
admittedly, there is some of that. But my intended point goes deeper. For key
bindings to work in medium-to-large applications two things are required: (1)
we need a lot of key binding real estate to find homes for all of the
operastions, and (2) the bindings need to be memorable. The typical binding
approach solves #1 by resorting to various modifier combos. But that causes #2
to crash. Modifiers are abstract entities, with no inherent meaning. Some of
the most important operations will get a good binding: eg, CMD-C for copy and
CMD-P for print are easy enough to remember. But in this system, the prime
real estate (the best letters) go fast. The end result almost always tends to
be a mess.

Modes as a pattern. On the surface, this point is more vim-vs-emacs. But,
again, I want to go deeper. Vim ends up with easier to type and remember key
bindings than emacs because it decides to pay a small cost up front: modes.
It's a simple idea: in some contexts, pressing `x` might do some operation; in
a different context, `x` is still open real estate for your binding needs.
Yes, the user must get used to switching modes -- but those switches are also
easy to execute in a system where the main operations require nothing more
than single key presses. Many classic apps are even better candidates for
modal key bindings than text editors. Consider Excel. For a mainstream app its
bindings are OK: they can be memorized to increase your speed and efficiency.
But doing some takes some effort and some of the moves are awkward. Now
imagine Excel with modes. All sorts of navigational and cell-selection
maneuvers can now be done with mnemonic key presses, without any modifiers.
Yes, in those moments when you are actually performing data entry, you need to
switch modes. But as a former heavy Excel user I know that the majority of
time is spent on tasks other than direct data entry.

      What a memory researcher would notice about Vim is compositional
      grammar. You don't memorize d3w as an atom; you learn d (delete), 3
      (count), w (word) and the combination is inferrable. Cognitive
      psychology calls this chunking plus schema: a small set of learned rules
      generates a large number of derivable combinations. The schema does the
      memorization work for you. Emacs has no equivalent — each command is its
      own thing to learn.

Grammars as a pattern. How do you create a powerful, scalable, and mnemonic
binding scheme? By embracing a grammar. Vim does it: there are operations
(insert, append, delete, copy) and their are entities (text objects and spans
of text defined by motions). In other words, VERB then NOUN. LL v2 makes a
similar move. Its binding scheme -- and the often parallel menu scheme --
follow a NOUN then VERB approach. Under this strategy you need a letter for
each noun or entity in your system and you need a letter for the various
operations. Even though there are very few letters, you can cover a lot of
ground using 2-letter combos. In LL, nearly all of the letters have a strong
mmenonic tie (v=video, c=chapter, j=jump, d=delete). The most common or
meaningful operations get the double-letter bindings: cc, ss, ll, mm to create
chapters, sections, saved loops, marks. In some cases, non-mnemonic choices
were required: `s` was claimed by sections, so the scratch loop bindings had
to use `x` for their prefix. As long as the number of such moves is limited, a
typical person can memorize the scheme.

Multi-key bindings more powerful than chords. The modifier-free approach to
bindings works better than the chordal approach for several reasons. They
scale better: 26 letters x 26 letters = 676 bindings. Versus: 26 letters * 4
modifiers = 104. Yes, you can create more modifier combos, but at even greater
mental cost. Multi-key scheme can be connected directly to grammatical or
other thematic schemes, helping the binding system to be more memorable. The
bindings become a kind of language, with structure. Humans excel at learning
material like this. Modifier-based binding schemes tend to lack that kind of
structure: they quickly become long lists of things to memorize (computers are
good at that, not humans) rather than meaningful entities operating within
some kind of logical structure.

Not the Stephenson argument. The classic pro-keyboard arguments -- Stephenson,
Engelbart, the Unix fans -- mostly argue that keyboard-centric tools give more
power to expert users. True, but that is a different argument. My claim: a
well-designed modal, structured, mnemonic binding scheme can supply better UX
for both expert and casual users alike. The case does not rest on users
becoming power users. And I'm a realist: very few users will ever be power
users in any given tool.

Engelbart on the mouse. Engelbart invented the mouse as half of a two-part
system. The other half was a chorded keyboard -- a 5-key left-hand device
where simultaneous key combinations (like piano chords) produced characters
and commands. Right hand for pointing, left hand for commanding. The mouse was
adopted; the keyset was not. His insight about the mouse is exactly right: it
is the right tool for spatial and positional tasks -- selecting regions,
navigating 2D space, pointing at structure. His frustration was that the
keyset was abandoned, leaving users with only the pointing half of a system
designed as two-handed.

The language argument. Most people will never be Vim masters. Stephenson-style
arguments expect too much of humanity. But all humans are already expert at
something much harder than Vim: natural language. Language has grammar
(structure), vocabulary (entities and operations), and meaning baked together.
Humans absorb this not because they are especially smart but because the
system has coherent, meaningful structure that maps onto real things. A binding
scheme built around STRUCTURE / GRAMMAR / MEANING / MNEMONIC leverages the
same cognitive machinery that makes language learnable. The cognitive demands
of a well-designed binding system can be lower than the demands of learning
the app's features themselves -- which you have to learn regardless.

NOUN VERB vs. VERB NOUN. Vim: VERB NOUN (delete word, yank paragraph). LL
v2: NOUN VERB (select entity, then act on it). The LL order is closer to how
GUIs work -- select something, then do something to it. It maps onto a mental
model users already have, rather than requiring a new one. This probably
contributes to the low cognitive load of the LL binding system.

Cognitive stickiness. The "meaning-attached bindings are stickier" observation
has real cognitive science behind it. Depth-of-processing theory (Craik &
Lockhart, 1972): information encoded with semantic content is retained far
better than rote symbol-to-action pairs. Mnemonics work because they attach
arbitrary signals to meaning already in long-term memory. Modifier-based
schemes force rote learning. A structured, mnemonic scheme allows semantic
encoding. This is not just intuition about UX; it is the lab finding applied
to binding design. Observable at the personal level over decades of learning
apps: memorable bindings stick; bindings with no inherent logic can stick if
used constantly (you pay the tax); but meaning-attached bindings are
inherently and durably stickier.

Which-key as the bridge to casual users. Press a prefix, see the available
completions. Not documentation -- in-context revelation at the moment of
motivated need. That is exactly when learning sticks. A casual user who presses
`s` and sees a structured menu of section operations is being educated without
being asked to study. The system starts easy and progressively educates any
curious user as they use it, in meaningful bits, without opening any
documentation. This is the key mechanism for making a structured binding
scheme accessible below the expert end of the spectrum.

Qualifications and realism. Many users have limited needs. Many needs are
inherently limited. The LL help doc ends its quick start by saying "most of
you can stop reading." A casual user will click along happily. But even a
casual user will pick up core bindings if they use the tool frequently enough.
Everyone is a casual user somewhere. A long-time Vim devotee may barely touch
Powerpoint. A guitarist who uses LoopLlama daily may show no interest in
mastering Excel. The argument is not that everyone will internalize a full
binding system. It is that even occasional users will absorb some of a
well-designed system -- especially if which-key is there to help -- and that
even casual users have their domains. A heavy knitting-app user would benefit
from a thoughtful binding scheme if that culture had developed differently.

The ecosystem argument. The most structural point in the whole critique. The
failure of good binding design in mainstream apps is not simply that developers
made bad choices. The OS and the surrounding culture shaped what was possible
and expected. In the current ecosystem -- flat, modifier-based, culturally
entrenched -- even developers who care about keyboard ergonomics produce worse
binding schemes than they otherwise would. The path of least resistance leads
to menus and modifiers. If the major OS makers had developed a culture and
infrastructure oriented around structured, namespace-based binding systems,
even less conscientious developers would produce better schemes by default.
Individual counter-examples -- Vim, Emacs, LoopLlama -- exist within a bad
ecosystem and are swimming upstream. They do not move the median. The median
is set by the platform.

Two mechanisms for the local maximum. Why are we stuck? Not just user habit,
though that matters. The deeper lock-in is developer tooling, developer
expectations, and OS-level cultural norms -- all compounding over time. Each
generation of developers learns to build flat modifier-based schemes because
that is what the platform rewards and what users expect. The knitting app is
not built well not because its developers are lazy, but because the ecosystem
gives them no reason and no tools to do better.

Vim versus emacs: the real steelman. The classic vim arguments are
well-known. The emacs case deserves a genuine hearing. Its strongest
argument is cognitive, not ergonomic: modal editing requires tracking
an invisible variable at all times -- what mode am I in? -- which is a
real tax when editing is interleaved with reading, thinking, and
context-switching. The emacs model says there is only one mode; Ctrl
and Meta are the admission price. A second argument: emacs key bindings
are the readline defaults, meaning bash, zsh, the Python REPL, and most
terminal programs all use them. Learn emacs bindings and you get them
for free across the terminal ecosystem. A third: M-x gives every
function a name, making the editor self-documenting -- you can find a
feature without knowing its binding.

Reassessing the emacs case. On closer inspection, several of those
arguments are weaker than they appear. The readline ubiquity claim is
an argument about network effects, not design quality -- and it gets the
causal arrow backwards. Emacs key bindings predate readline; GNU
readline was explicitly designed to replicate emacs-style line editing
in bash. "Emacs bindings are everywhere" is a consequence of emacs
influence, not an independent reason to prefer the model. The Lisp
extensibility argument and the discoverability argument are real
advantages -- but both are orthogonal to key binding design. Vim could
have had a better scripting environment (and now has one, via Lua).
Better discoverability infrastructure could have been built on top of
vim long ago, and increasingly has been. These are tooling gaps, not
structural features of the binding model.

Emacs' partial grammar. Emacs bindings do have some structure. The
clearest pattern is Meta-as-amplifier: Ctrl-f moves one character,
Meta-f moves one word; Ctrl-d deletes a character, Meta-d deletes a
word; Ctrl-k kills to end of line, Meta-k kills to end of sentence.
That is a genuine two-level grammar where the modifier encodes scope.
Prefix namespaces -- Ctrl-x for file and buffer operations, Ctrl-c for
mode-local bindings -- impose further organization. But the structure is
shallower than vim's generative grammar: more lookup table than
language. And the prefix namespaces are themselves an implicit
concession that flat chordal bindings do not scale.

Emacs halfway to wisdom. The Ctrl-x completion display -- press the
prefix, see all available continuations -- is essentially the which-key
idea. When emacs advocates cite this as a discoverability strength, they
are endorsing exactly the principle that makes structured prefix
namespaces good: enumerable, browsable completions revealed at the
moment of need. The insight points directly toward the vim leader-key
model, where the main command namespace requires no modifier at all. But
emacs cannot escape its foundation. Ctrl is load-bearing all the way
down. The right idea was found; the original sin was already baked in
too deep to undo.

Vim's grammar and the which-key objection. One common claim: vim's
generative verb-motion grammar is structurally resistant to which-key
style help. After `d`, the valid completions are "any motion" -- a
semantic category, not a finite list. But this argument is overstated.
The actual usage distribution is heavily concentrated. A curated display
of the 25 or 30 most common motions and text objects in operator-pending
mode would cover the vast majority of real use. Exhaustiveness is not
the standard; useful coverage is. An additional keypress could surface a
paginated reference -- exactly how well-designed CLI help works. The
gap is tooling culture, not structural impossibility. Vim's culture
skewed toward RTFM and muscle memory for long enough that the
infrastructure was never built. It can be.

Vim-style spreadsheets. Spreadsheets are arguably a better fit for
modal editing than text editors. The majority of time in a spreadsheet
is not data entry -- it is navigation, selection, formatting, review,
formula inspection. That is exactly the work that modal single-key
commands handle well: insert mode for cell entry, everything else driven
by mnemonic key presses, no modifiers required. This was noticed. sc
(Spreadsheet Calculator), a Unix terminal program from the early 1980s,
had hjkl navigation from the start. Its modern fork, sc-im, has genuine
modal editing: normal mode for navigation and commands, insert mode for
cell entry, a vim-style command line. VisiData is a related terminal
tool for tabular data exploration with vim-inspired navigation -- not a
formula-driven spreadsheet, but adjacent in spirit. The mainstream never
picked this up. Classic Excel's own model was implicitly modal --
navigation state versus cell-edit state, the latter triggered by F2 or
typing -- but the abstraction was never made explicit and never exploited
for keybinding design. The extensive keyboard support in classic Windows
Excel was genuinely powerful, arriving through a patchwork of special
keys rather than a coherent modal scheme. A near miss.
